# 回答文作成システム 要件定義書

**作成日**: 2026-03-17
**ステータス**: ドラフト

---

## 1. プロジェクト概要

### 1.1 背景
商品に問題があった場合、原因調査を行い、その結果をお客様へ回答する業務が存在する。
現状はExcelベースのマニュアル集から適切な回答パターンを手動で選択し、回答文を作成している。
このプロセスをシステム化し、正確性の向上と作業効率の改善を図る。

### 1.2 現状の課題
- マニュアル集から適切な回答パターンを探す手間がかかる
- 条件に応じた追記の漏れリスクがある
- 回答の品質が担当者のスキルに依存する
- 回答履歴の管理・検索が困難

### 1.3 ゴール
条件を選択・入力することで、適切な回答文を自動生成するシステムを構築する。

---

## 2. 業務フロー

```
[問題発生] → [原因調査] → [システムで分類選択] → [回答文自動生成] → [確認・編集] → [お客様へ送付]
```

### 2.1 ユーザー操作フロー
1. 容器の種類を選択
2. 問題の大分類を選択（例: 漏れ・破損）
3. 中分類を選択（例: 亀裂） → **主回答文が決まる**
4. 小分類を選択（例: 小さな傷） → **サブ回答文が決まる**
5. サブ回答文の付与条件を確認（Yes/No形式）
6. 回答文（調査結果/製造工程/まとめ）を自動生成
7. ユーザーが確認・必要に応じて編集
8. 最終成果物を出力

---

## 3. 機能要件

### 3.1 マスタ管理機能
- **容器マスタ**: 容器の種類を管理（6〜20種類、追加可能）
- **問題分類マスタ**: 大分類 → 中分類の2階層で管理
  - 中分類ラベルごとに主回答文テンプレートが紐づく
- **小分類マスタ**: 中分類ごとに別テーブルで管理
  - 各中分類の小分類は独立したテーブルを持つ（情報量の分離）
  - 例: 亀裂の小分類テーブル → 小さな傷 / 大きな傷 / 裂傷 / 押し込み
- **主回答文テンプレート**: 中分類レベルで決まるベース回答文
- **サブ回答文テンプレート**: 小分類や追加条件に応じて主回答文の前後に付与するテキスト

### 3.2 回答文生成機能
- 容器×中分類から主回答文を自動特定
- 小分類の選択に応じてサブ回答文を付与
- 3パート構成の回答文を生成:
  - **調査結果**: 調査内容と判明した原因
  - **製造工程**: 関連する製造工程の説明
  - **まとめ**: 結論と今後の対応
- サブ回答文は主回答文の前または後ろに結合するだけのシンプルな仕組み
- 生成後の手動編集機能

### 3.3 出力機能
- 画面上でのプレビュー
- ファイル出力（形式は後続フェーズで決定）

### 3.4 履歴管理機能
- 過去の回答文を検索・参照
- どの分類・条件で生成されたかの記録

## 4. DB設計（コア）

### 4.1 ER図（概念）

```
[containers]             [problem_categories]
容器マスタ                 問題分類マスタ（大分類→中分類）
 |                            |
 +----------------------------+
               |
        [manual_entries]
        マニュアルエントリ
        （容器×中分類 → 主回答文への紐付け）
               |
        [response_templates]
        主回答文テンプレート
        （調査結果/製造工程/まとめ の3パート）
               |
        [sub_responses]
        サブ回答文テンプレート
        （小分類や条件に応じて主回答文の前or後に付与）


[crack_types]  [leak_mark_types]  [lid_crack_types]  ...
亀裂の小分類    漏れ跡の小分類      缶蓋亀裂の小分類
  ↑ 中分類ごとに独立したテーブル（情報量の分離）


[response_history]
回答履歴
```

### 4.2 テーブル定義

#### 4.2.1 containers（容器マスタ）
| カラム名 | 型 | 説明 |
|---------|------|------|
| id | INT PK | 容器ID |
| name | VARCHAR | 容器名（例: 缶、ボトル、パウチ） |
| description | TEXT | 備考 |
| sort_order | INT | 表示順 |
| is_active | BOOLEAN | 有効フラグ |
| created_at | TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### 4.2.2 problem_categories（問題分類マスタ）
大分類→中分類の2階層。中分類が主回答文の特定に使われる。

| カラム名 | 型 | 説明 |
|---------|------|------|
| id | INT PK | 分類ID |
| parent_id | INT FK(self) | 親分類ID（NULLなら大分類） |
| depth | INT | 階層の深さ（1=大分類, 2=中分類） |
| name | VARCHAR | 分類名 |
| detail_table_name | VARCHAR NULL | 小分類テーブル名（中分類のみ。例: "crack_types"） |
| description | TEXT | 備考 |
| sort_order | INT | 同階層内の表示順 |
| is_active | BOOLEAN | 有効フラグ |
| created_at | TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### 4.2.3 小分類テーブル群（中分類ごとに独立）
中分類ごとに専用テーブルを作成する。各テーブルは同一構造。

**例: crack_types（亀裂の小分類）**

| カラム名 | 型 | 説明 |
|---------|------|------|
| id | INT PK | 小分類ID |
| name | VARCHAR | 小分類名（例: 小さな傷、大きな傷、裂傷、押し込み） |
| description | TEXT | 備考 |
| sort_order | INT | 表示順 |
| is_active | BOOLEAN | 有効フラグ |
| created_at | TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | 更新日時 |

**同様に作成するテーブル:**
- `leak_mark_types`（漏れ跡の小分類）
- `lid_crack_types`（缶蓋亀裂の小分類）
- その他、中分類が追加されるたびに対応テーブルを作成

#### 4.2.4 manual_entries（マニュアルエントリ）
容器×中分類の組み合わせを管理するハブテーブル。主回答文の特定に使う。

| カラム名 | 型 | 説明 |
|---------|------|------|
| id | INT PK | エントリID |
| container_id | INT FK | 容器ID |
| category_id | INT FK | 問題分類ID（中分類を指定） |
| name | VARCHAR | エントリ名（管理用） |
| description | TEXT | 備考 |
| is_active | BOOLEAN | 有効フラグ |
| created_at | TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### 4.2.5 response_templates（主回答文テンプレート）
各パートのテンプレート本文を管理。

| カラム名 | 型 | 説明 |
|---------|------|------|
| id | INT PK | テンプレートID |
| manual_entry_id | INT FK | マニュアルエントリID |
| section_type | ENUM | パート種別: 'investigation' / 'process' / 'summary' |
| body | TEXT | テンプレート本文（プレースホルダ対応） |
| sort_order | INT | 同セクション内の表示順 |
| is_active | BOOLEAN | 有効フラグ |
| created_at | TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### 4.2.6 sub_responses（サブ回答文テンプレート）
小分類や追加条件に応じて主回答文の前または後ろに付与するテキストを管理。

| カラム名 | 型 | 説明 |
|---------|------|------|
| id | INT PK | サブ回答文ID |
| template_id | INT FK | 対象の主回答文テンプレートID |
| detail_type | VARCHAR | 紐づく小分類テーブル名（例: "crack_types"） |
| detail_id | INT NULL | 小分類ID（NULLなら小分類に依存しない追加条件） |
| condition_label | VARCHAR NULL | 条件の表示ラベル（例: "流通経路による場合"）。detail_idで決まる場合は不要 |
| position | ENUM | 付与位置: 'before' / 'after' |
| body | TEXT | サブ回答文の本文 |
| sort_order | INT | 同位置内の表示順 |
| is_active | BOOLEAN | 有効フラグ |
| created_at | TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### 4.2.7 response_history（回答履歴）
生成された回答の履歴を記録。

| カラム名 | 型 | 説明 |
|---------|------|------|
| id | INT PK | 履歴ID |
| manual_entry_id | INT FK | 使用したエントリID |
| container_id | INT FK | 容器ID |
| category_id | INT FK | 問題分類ID（中分類） |
| detail_type | VARCHAR | 小分類テーブル名 |
| detail_id | INT | 選択した小分類ID |
| generated_text | TEXT | 生成された回答文全文 |
| edited_text | TEXT NULL | 編集後の回答文（編集があった場合） |
| created_by | VARCHAR | 作成者 |
| created_at | TIMESTAMP | 作成日時 |

#### 4.2.8 response_history_subs（履歴×適用したサブ回答文）
履歴にどのサブ回答文が適用されたかを記録。

| カラム名 | 型 | 説明 |
|---------|------|------|
| id | INT PK | ID |
| response_history_id | INT FK | 履歴ID |
| sub_response_id | INT FK | 適用したサブ回答文ID |

| id | template_id | detail_type | detail_id | condition_label | position | body |
|----|------------|-------------|-----------|----------------|----------|------|
| 1 | 1 | crack_types | 1 (小さな傷) | NULL | after | なお、傷は微小であり、内容物への影響はございません。 |
| 2 | 1 | crack_types | 2 (大きな傷) | NULL | after | 傷の深さから内容物への影響が懸念されたため、外部検査機関による分析を実施いたしました。 |

| 4 | 1 | NULL | NULL | 流通経路による場合 | after | 流通過程における取り扱い状況を確認いたしましたところ、輸送時の振動により缶体に負荷がかかった可能性が認められました。 |

**動作の仕組み:**

- `detail_id` が指定されている → 小分類の選択で**自動的に**付与される
- `detail_id` がNULLで `condition_label` がある → ユーザーに**Yes/Noで質問**して付与

### 5.7 生成される回答文の例

**入力:** 350ml缶 × 漏れ・破損 → 亀裂 → **小さな傷**

**追加質問:** 「流通経路による場合」→ **はい** / 「当たったものが明確な場合」→ いいえ

---

> **【調査結果】**
> お届けいたしました緑茶「まるごと茶葉」に傷が確認された件につきまして、調査結果をご報告いたします。現品を確認したところ、缶体表面に傷が認められました。傷の形状および方向から、製造ラインの搬送過程において缶同士の接触により生じたものと判断いたしました。なお、傷は微小であり、内容物への影響はございません。流通過程における取り扱い状況を確認いたしましたところ、輸送時の振動により缶体に負荷がかかった可能性が認められました。
>
> **【製造工程】**
> なお、本件は製造工程内ではなく、流通過程で生じた可能性が高いと判断しております。当該製品の製造工程は以下の通りです。缶体成形 → 洗浄 → 充填 → 密封 → 殺菌 → 冷却 → 外観検査 → 梱包。外観検査工程ではカメラによる自動検査を実施しております。
>
> **【まとめ】**
> このたびはご不快な思いをおかけしましたことを深くお詫び申し上げます。搬送ラインのガイド調整を実施し、缶同士の接触を低減する対策を講じました。

**適用されたサブ回答文:**
- サブ1（小さな傷 → 自動付与）→ 調査結果の後ろに「傷は微小であり〜」
- サブ4（流通経路 → Yes）→ 調査結果の後ろに「流通過程における〜」
- サブ5（流通経路 → Yes）→ 製造工程の前に「なお、本件は〜」
- サブ7（当たったもの → No）→ 付与されない

### 5.8 データの流れまとめ

```
ユーザーが選択:
  容器:     [350ml缶]         → containers.id = 1
  大分類:   [漏れ・破損]       → problem_categories.id = 4
  中分類:   [亀裂]             → problem_categories.id = 40
                                  → 主回答文が決まる
  小分類:   [小さな傷]         → crack_types.id = 1
                                  → 該当サブ回答文が自動付与

        ↓ マッチング

  manual_entries: id=1（缶×亀裂）が該当

        ↓ 主回答文取得

  response_templates: エントリ1に紐づく3パート取得

        ↓ サブ回答文取得

  sub_responses:
    detail_id=1（小さな傷）のサブ → 自動付与
    detail_id=NULLのサブ → ユーザーにYes/No質問

        ↓ 主回答文 + サブ回答文(前or後)を結合 + プレースホルダ置換

  最終回答文を生成
```

## 6. DB設計のポイント

### 6.1 問題分類の2層構造
- `problem_categories` は大分類→中分類の2階層のみ
- 中分類が主回答文の特定単位。中分類まで選べば主回答文が決まる

### 6.2 小分類テーブルの分離
- 中分類ごとに独立した小分類テーブルを持つ（`crack_types`, `leak_mark_types` 等）
- 1テーブルに全分類を詰め込まず、情報量を分離
- `problem_categories.detail_table_name` で対応する小分類テーブルを参照
- 中分類の追加時に新しい小分類テーブルを作成する運用

### 6.3 サブ回答文の2種類の付与方式
- **自動付与**: `detail_id` が指定されている → 小分類の選択で自動的に主回答文に結合
- **質問付与**: `detail_id` がNULL → `condition_label` を使ってユーザーにYes/No質問
- いずれも `position`（before/after）で主回答文の前か後ろかを指定するだけ

### 6.4 テンプレートの構成
- `response_templates` で3パート（調査結果/製造工程/まとめ）を個別管理
- 本文にプレースホルダ（例: `{{product_name}}`）を埋め込み、動的に置換

## 9. 用語集

| 用語 | 意味 |
|------|------|
| 容器 | 商品の容器種別（缶、ボトル等） |
| 問題分類 | 大分類→中分類の2階層で管理する問題点の分類体系 |
| 小分類 | 中分類ごとに独立テーブルで管理する詳細分類（亀裂→小さな傷等） |
| マニュアルエントリ | 容器×中分類の組み合わせに対応するマニュアル項目 |
| 主回答文 | 中分類レベルで決まるベースの回答文（プレースホルダを含む） |
| サブ回答文 | 小分類や追加条件に応じて主回答文の前後に付与するテキストブロック |
| パート | 回答文の構成要素（調査結果/製造工程/まとめ） |
